Web UI

The web interface has 5 sections:

  • Hypotheses: allows you to suspect possible causes of failure, and later confirm them.
  • Actions: supplements the RViz UI in helping you either to find out what's wrong or to correct the failure.
  • Task: serves two functions
    • It showcases the error information in the current task hierarchy (more on the hierarchy below).
    • It presents buttons to allow you to choose how to resume the task (please check in with me before resuming execution).
  • Faults: reports the status of component-focused background fault monitors.
  • Beliefs: reports the robot's knowledge of the task, the environment, or itself; represented as boolean predicates.

Note: If a request for help appears shortly after you have already completed one (within seconds), please refresh the page.

Hypotheses
  • You can suspect (select) possible causes of error using the dropdowns.
  • You need to suspect at least 1 hypothesis in order to take actions.
  • You can confirm suspected hypotheses using the checkbox.
  • You need to confirm at least 1 hypothesis in order to resume the task.

Faults & Beliefs

Fault monitors are background monitors that periodically check the status of robot components:

  • If there is a fault in the component, the monitor status show fault
  • If there is no error, the monitor status shows nominal

Beliefs are boolean predicates that capture the robot's knowledge of the task, the environment, or itself:

  • A belief can evaluate to true
  • A belief can evaluate to false
  • If there is not enough information for a belief, such as if it is irrelevant to the task or environment at hand, then the belief can evaluate to none
Actions
  • The results of executing actions are displayed in the Status section. Some actions return additional details in their results that can inform your next steps.
  • For safety reasons, please be aware:
    • Base movements do not perform collision checking.
    • Please read the descriptions of actions before executing them because actions can have consequences that are a surprise to some.
  • Generally, please wait for an action to return a result before issuing another one. If in doubt, ask me.

Task

Tasks are specified as a hierarchy of subtasks and primitive actions that are executed in sequence. An example (primitive actions are coloured in yellow):

Consider a failure in the arm action (press the buttons):

On a failure, we show:

  • The name of the task
  • The last 15 primitive actions that were executed; regardless of whether they were successful or not.
  • Details of the subtasks and actions in the currently executing task.

For each of the active levels in the hierarchy, we show:

  • The number of times the subtask or action has failed.
  • The position (index) of the child subtask or action that was being executed.
  • The parameters that were sent to each of the subtasks

Note: the information in the hierarchy displays what the robot should execute, and not necessarily what it actually executed.

When you are ready to resume from a failure, the buttons in the UI allow you to specify where in this hierarchy to resume. Since you might not be familiar with the full task hierarchy, check with me before clicking on a button to resume.
